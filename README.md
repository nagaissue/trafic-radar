## 📌 概要

北海道の公共交通（JR北海道・路線バス等）は、天候や事故により遅延・運休が頻発しますが、情報が路線ごとに分散しており、
利用者がリアルタイムに全体像を把握しづらいという課題があります。

本プロジェクトは、公開されている運行データ（GTFS / GTFS-Realtime など）を定期的に取得・集約し、
利用者が登録した路線の遅延情報をダッシュボードで確認・通知として受け取れる仕組みを提供します。

交通インフラ領域への関心から着想したテーマであり、実際の業務システム開発（データ収集→非同期処理→通知→可視化）を
一通り経験できる設計を意識しています。

## ✨ 主な機能

各機能について「何のためにあるか」「どう動くか」「技術的に何を使うか」「実装時の注意点」をまとめています。
チーム内での実装分担・認識合わせの際はこのセクションを参照してください。

### 1. 運行情報の自動収集

**何のための機能か**
アプリの土台となるデータ収集機能。GTFS(静的)とGTFS-RT(動的)の2種類のデータを定期的に取得し、
DBに蓄積することで、他の全機能（ダッシュボード表示・通知）の材料を作る。

**主な処理の流れ**
1. **GTFS(静的)取得**：バス会社が配布するZIPファイルを定期取得（頻度低め、例：1日1回）
   - `stops.txt`（停留所）、`routes.txt`（路線）、`trips.txt`（便）、`stop_times.txt`（時刻表）を解析
   - 内容が更新されていればDBの路線マスタを更新
2. **GTFS-RT(動的)取得**：Protocol Buffers形式のリアルタイムデータを高頻度取得（例：1〜5分ごと）
   - `gtfs-realtime-bindings` ライブラリでデコード
   - 「遅延」「運休」などのステータス情報を抽出
3. **非同期処理へ引き渡し**：取得したデータをそのままDB保存するのではなく、SQSへ投入し
   「解析→通知判定→送信」を後続処理として分離する（詳細は「3. ユーザー登録路線の通知」参照）

**技術要素**：Spring Scheduler（定期取得）／ gtfs-realtime-bindings（GTFS-RTデコード）／ Amazon SQS（処理分離）／ Flyway（スキーマ管理）

**実装時の注意点**
- GTFS-RTは生データのままでは読めない（Protocol Buffers形式）ため、最初は「デコードして中身を見る」だけの検証コードを書くと理解が早い
- バス会社によってGTFSの提供形式・更新頻度が異なる可能性があるため、最初に触るデータソースを1つに絞って検証する
- JR北海道は構造化データが公開されていない可能性があり、スクレイピングが必要になる場合がある（要調査、規約確認必須）

### 2. 路線・区間の遅延ダッシュボード

**何のための機能か**
収集した運行情報を、ユーザーが直感的に把握できる形で見せる画面。本アプリの「顔」となる機能で、
就活のデモでも一番見せる部分になる想定。

**主な処理の流れ**
1. フロントエンドがバックエンドAPIに現在の遅延状況を問い合わせる（TanStack Queryでポーリング）
2. バックエンドはDBから最新の遅延情報を取得し、REST APIとして返却
3. フロントエンドは以下の2形式で表示
   - **地図表示**：MapLibre GL JS + react-map-gl で路線・停留所と遅延状況を重ねて表示
   - **一覧表示**：路線ごとの遅延状況をテーブル形式で表示（地図が苦手な人向けの代替導線）

**技術要素**：TanStack Query（ポーリング）／ MapLibre GL JS・react-map-gl（地図表示）／ Tailwind CSS + shadcn/ui（UI構築）／ openapi-typescript（型自動生成）

**実装時の注意点**
- ポーリング間隔は「リアルタイム感」と「サーバー負荷」のトレードオフなので、最初は30秒〜1分程度から調整する
- 地図表示は学習コストが高いため、まず一覧表示だけで動く状態を作ってから地図表示に着手する順番がおすすめ
- 停留所・路線の位置情報（緯度経度）はGTFSの `stops.txt` に含まれるため、地図表示前にデータの中身を確認しておく

### 3. ユーザー登録路線の通知

**何のための機能か**
ユーザーが「よく使う路線」を登録しておくと、その路線に遅延が発生した際に能動的に知らせてくれる機能。
このアプリの中で唯一「非同期処理」の設計・実装を本格的に経験できるパート。

**主な処理の流れ**
1. GTFS-RTから遅延情報を検知（「1. 運行情報の自動収集」の続き）
2. 検知した遅延情報をトリガーに、以下の判定処理を行う
   - どのユーザーがこの路線を登録しているか（DB照会）
   - 通知条件を満たしているか（例：〇分以上の遅延のみ通知、等）
3. 通知対象が確定したら、送信処理を実行
   - メール：Amazon SES
   - チャット：LINE Messaging API（LINE Notifyはサービス終了済みのため採用）

**段階的な実装方針**（最初から本格構成にすると学習コストが高いため）

| ステップ | 内容 |
|---|---|
| ステップ1（学習用） | Spring Event（`ApplicationEventPublisher`）を使い、同一プロセス内でのイベント駆動処理を体験する |
| ステップ2（本格構成） | Amazon SQSを導入し、「取得」「解析」「通知判定」「送信」を別々の処理として分離する |

**技術要素**：Spring Event（学習用）／ Amazon SQS（本格構成）／ Amazon SES（メール）／ LINE Messaging API（チャット）／ WireMock（テスト用モック）

**実装時の注意点**
- 「誰にでも全部通知する」ではなく「登録した路線だけ通知する」設計にするため、ユーザーと路線の中間テーブル（お気に入り）の設計が重要
- 通知の二重送信を防ぐため、「一度通知した遅延情報には再送しない」ようなステータス管理が必要になる
- LINE Messaging APIはBotとの友だち登録がユーザー側に必要になるため、UI上で案内文言を用意する

### 4. 認証・認可

**何のための機能か**
「誰がログインしているか」「そのユーザーが何を編集していいか」を管理する基盤機能。
他の全機能（お気に入り登録、管理者機能）の前提となる。

**主な処理の流れ**
1. ユーザー登録・ログイン画面からID/パスワードを送信
2. バックエンドで認証を行い、JWT（JSON Web Token）を発行
3. フロントエンドはJWTを保持し、以降のAPIリクエストに付与
4. バックエンドはJWTを検証し、リクエストごとに「誰のリクエストか」「権限があるか」を判定

**権限設計の考え方**

| ロール | できること |
|---|---|
| 一般ユーザー | 自分のお気に入り路線の登録・編集・削除のみ可能 |
| 管理者 | データソースの管理、通知ルールの設定など全体管理が可能（「5. 管理者機能」参照） |

**技術要素**：Spring Security（認証・認可基盤）／ JWT（ログイン状態の保持）

**実装時の注意点**
- 「自分のデータしか編集できない」権限チェックを実装で漏らしやすいので、Controller層ではなくService層で一貫してチェックする設計にする
- JWTの有効期限切れ時の挙動（自動ログアウト、リフレッシュトークンの要否）を最初に決めておくと後の手戻りが少ない

### 5. 管理者機能

**何のための機能か**
データソース（GTFS取得元）や通知ルールなど、システム全体の設定を管理する機能。
一般ユーザーには見せない、運用者向けの画面。

**主な処理の流れ**
1. 管理者権限でログイン（「4. 認証・認可」の権限チェックに依存）
2. データソース管理画面から、取得対象のバス会社・取得間隔などを確認・変更
3. 通知ルール管理画面から、通知条件（何分以上の遅延で通知するか等）を設定

**技術要素**：Spring Security（ロールベース制御）／ Spring Data JPA（設定値の永続化）

**実装時の注意点**
- 優先度は他機能より低め。まずは一般ユーザー向け機能（1〜4）を動く状態にしてから着手する
- 最初は「設定ファイルで固定値を持つ」レベルからスタートし、余力があれば管理画面から変更できるようにする、という段階的な実装でもよい

## 🔗 機能間の関係図

```
[1. 運行情報の自動収集]
        │ 遅延情報を検知
        ▼
[3. ユーザー登録路線の通知] ──── 通知対象を判定 ──── [4. 認証・認可]
        │ 蓄積データを参照                                  │ ユーザー・権限情報を提供
        ▼                                                    ▼
[2. 路線・区間の遅延ダッシュボード]                  [5. 管理者機能]
        （全ユーザーが閲覧）                        （管理者のみ設定変更）
```

**実装順の目安**：`4.認証` → `1.データ収集` → `2.ダッシュボード` → `3.通知` → `5.管理者機能`
（認証は他機能の土台のため最優先、管理者機能は優先度が低いため最後）

## 🏗 アーキテクチャ概要

```
[GTFS(静的ZIP)]        [GTFS-RT(Protocol Buffers)]      [JR北海道 運行情報]
      │                         │                              │
      └──────────────┬──────────┴──────────────────────────────┘
                      ▼
            [取得バッチ (Spring Scheduler)]
                      │
                      ▼
             [SQS] ─ 解析 → 通知判定 → 送信 を分離
                      │
                      ▼
          [Spring Boot API Server] ── [RDS(PostgreSQL)]
                      │  REST API (OpenAPI)
                      ▼
        [React + TypeScript SPA (Vite)]
                      │
        ┌─────────────┴─────────────┐
        ▼                           ▼
 [MapLibre 地図表示]         [TanStack Query ポーリング]

通知経路: SQS → 通知判定 →  SES(メール) / LINE Messaging API

本番想定: CloudFront → ALB → ECS(Fargate) → RDS
                              └→ S3(静的アセット) / SQS / EventBridge
```

## 🏁 まずやることリスト（スプリント0：着手準備）

設計より先に「本物のデータの形」を知っておくことが重要なため、DB設計より前にGTFSデータに触れる工程を先頭に置いています。

### 1. 環境構築
- [ ] JDK 21 インストール
- [ ] IntelliJ IDEA（or 使い慣れたIDE）セットアップ
- [ ] Docker Desktop インストール
- [ ] Node.js（LTS版）インストール（React用）
- [ ] GitHubリポジトリ作成（backend/frontendはmonorepoでも別リポジトリでもOK、まずはmonorepoが管理楽）

### 2. GTFSデータを実際に触ってみる
- [ ] 北海道中央バス or 道北バス or 拓殖バスのGTFS(静的)データをダウンロード
- [ ] ZIPを解凍してCSVの中身を確認（`stops.txt`, `routes.txt`, `trips.txt`, `stop_times.txt` など）
- [ ] GTFS-RT(動的)データも実際に1回取得してみる（Protocol Buffers形式なのでそのままでは読めないことを体感する）
- [ ] どのデータを自分のDBに落とし込むか、ざっくりメモする

### 3. Spring Bootプロジェクトの雛形作成
- [ ] Spring Initializr（https://start.spring.io）で雛形生成
  - 依存関係：Spring Web, Spring Data JPA, PostgreSQL Driver, Flyway, Spring Security, Validation
- [ ] ローカルで `./gradlew bootRun` して起動確認だけする（中身は空でOK）

### 4. Docker Compose環境作成
- [ ] PostgreSQLコンテナを立てる `docker-compose.yml` を書く
- [ ] Spring BootからそのPostgreSQLに接続できることを確認

### 5. Reactプロジェクトの雛形作成
- [ ] `npm create vite@latest` でReact+TypeScriptの雛形作成
- [ ] Tailwind CSSのセットアップ
- [ ] 適当なページを1つ表示してみて動作確認

### 6. プロダクトバックログ作成
- [ ] GitHub Projectsでカンバンボード作成
- [ ] ユーザーストーリーをIssueとして書き出す（主な機能の5項目をベースに）

### 7. ER図の設計
- [ ] 路線・停留所・便・遅延ログ・ユーザー・お気に入りのテーブルをざっくり書く
- [ ] draw.io等で図にする

## 🚀 セットアップ手順（ローカル開発）

### 前提
- Docker / Docker Compose
- Java 21
- Node.js（LTS）

### 手順
```bash
# リポジトリをクローン
git clone https://github.com/<org>/transit-watch.git
cd transit-watch

# DB・関連サービスをコンテナで起動
docker compose up -d

# バックエンド起動（Flywayマイグレーションは起動時に自動適用）
cd backend
./gradlew bootRun

# フロントエンド起動（別ターミナル）
cd frontend
npm install
npm run dev
```
